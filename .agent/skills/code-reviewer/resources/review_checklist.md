# コードレビュー・チェックリスト

レビュー時は以下の観点を必ず確認してください。
本チェックリストは Swift/iOS 開発を主としつつ、他言語でも共通して適用できる観点です。

## 🛡️ 安全性 (Security)

- パスワードや API キーがソースコード内にハードコーディングされていないか？
- 機密情報（トークン、金額データ、個人情報等）を安全でない場所（`UserDefaults` 等）に平文で保存していないか？
  - iOS の場合: `Keychain` を使用すること。
- 入力値の検証（バリデーション）は適切か？（金額入力の負数チェック、文字列長の上限など）
- 外部からの入力をそのまま処理に渡していないか？（インジェクション系の脆弱性）
- ログ出力に個人情報や機密データが含まれていないか？

## 🏗️ アーキテクチャ (Architecture)

- MVVM パターンに従っているか？
  - View にビジネスロジックや複雑な分岐が含まれていないか？
  - 状態管理は ViewModel（`@Observable`, `@ObservableObject` 等）に委譲されているか？
- 単一責任の原則を守っているか？（1つのクラス・関数が複数の責務を持っていないか）
- 不要なフレームワークの import がないか？（SwiftUI プロジェクトでの不要な UIKit import 等）

## 🚀 パフォーマンス (Performance)

- ループの中で重い処理（ネットワーク通信、DB アクセス、ファイル I/O 等）を行っていないか？
- 無駄なオブジェクト生成やメモリアロケーションはないか？
- iOS 固有: メインスレッドで時間のかかる処理をブロックしていないか？（UI のフリーズにつながる）
- 不要な再描画・再計算が発生するコードになっていないか？（SwiftUI の `@State` / `@Binding` の過剰な更新等）

## 📖 可読性・保守性 (Readability)

- 変数名・関数名は「何をするか」を明確に表しているか？（`data`, `temp`, `d` などの曖昧な名前は NG）
- 複雑なロジックには「なぜそうしているか」のコメントがついているか？
- 関数が長すぎないか？（目安: 1関数 30行以内、超える場合は分割を検討）
- エラーハンドリングは適切か？（エラーを握りつぶしていないか）
  - Swift の場合: `catch {}` で空にしていないか？`do-catch` 内で適切にエラーを処理しているか？
- マジックナンバーや意味不明な定数を直接使っていないか？（定数として定義すべき）

## 🧪 テスト (Testing)

- エッジケース（境界値、`nil`、空の配列・文字列など）は考慮されているか？
- テストコードが実装より先に書かれているか？（TDD の原則）
- テスト対象の関数が外部依存（DB、ネットワーク等）から分離されているか？（モック化可能か）
