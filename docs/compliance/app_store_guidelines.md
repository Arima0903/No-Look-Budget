# App Store 審査ガイドライン対策・リスク管理

初めてのApp Storeリリースに向けて、審査（App Review）でリジェクト（却下）されやすいポイントと、Orbit Budgetにおける具体的な対策をまとめたドキュメントです。
開発の進行に伴い、このドキュメントを都度更新し、審査時のリスクを最小限に抑えます。

## 1. ビジネスモデル・課金関連 (Guideline 3.0)

### 🚨 リスクと懸念点
* 今後のマネタイズとして「広告非表示」や「強制ロック（罰金）」などのサブスクリプションを追加予定ですが、Appleは**「理解しにくい課金モデル」や「詐欺的なサブスクリプション」を厳しく審査**します。
* 特に「罰金」モデル等の突飛な課金は、ユーザー保護の観点からリジェクトされる可能性が高いです。

### 🛡 対策方針
* **MVPリリース時**: まずは完全無料アプリとしてリリースし、コア機能（ウィジェット、立替セパレーター）の動作を証明します。（これにより課金周りの審査リスクを一旦ゼロにする）
* **マネタイズ導入時**: 課金画面（Paywall）にて、「何が提供されるのか」「いくらかかるのか」「キャンセル方法は何か」を明瞭に記載したUIを設計します。
* 尖った課金（罰金など）は避け、まずはスタンダードな「プレミアム機能のアンロック」から開始します。

## 2. デザイン・最小限の機能 (Guideline 4.2 - Minimum Functionality)

### 🚨 リスクと懸念点
* Appleは「Webサイトでもできるような単純なアプリ」や「機能が少なすぎるアプリ」を「Minimum Functionality（最小限の機能要件を満たしていない）」としてリジェクトすることが多々あります。
* Orbit Budgetは「引き算のデザイン」をコンセプトにしていますが、審査員に「ただの残高表示アプリ（機能不足）」と判定されるリスクがあります。

### 🛡 対策方針
* **iOSネイティブ機能の活用**: アピールポイントとして「iOS 18 の WidgetKit / Live Activities」をフル活用していることを強調します。「この機能はネイティブアプリ（iOSのウィジェット）でしか実現できない体験である」ことを設計で示します。
* **Apple Wallet連携 (PoC対象)**: もし実装できれば、「システムレベルでの決済フック」は強力なネイティブ機能の証明になります。

## 3. プライバシーとデータの取り扱い (Guideline 5.1)

### 🚨 リスクと懸念点
* アプリが「お金」というセンシティブなデータを扱うため、データの保存場所やプライバシーポリシーの有無が厳しくチェックされます。
* 万が一、外部サーバーにユーザーの家計データを送るような設計にする場合、厳格な同意フローとセキュリティ基準が求められます。

### 🛡 対策方針
* **完全ローカル完結**: Orbit Budgetの全データはデバイス内部（`SwiftData` 等）にのみ保存し、**外部サーバー（Cloud）へは一切送信しない仕様**とします。（バックアップ目的の場合は、ユーザー自身の個人のiCloud領域のみを使用）
* 審査提出前に、「ユーザーデータを一切収集・送信しない」旨を記載したシンプルなプライバシーポリシーページ（WebサイトまたはNotion等の公開ページ）を作成し、提出時にURLを含めます。

## 4. 知的財産・著作権・商標 (Guideline 5.2)

### 🚨 リスクと懸念点
* 他社のサービス名（例: マネーフォワード、Zaim）、アイコン、ロゴなどをアプリ内やストアのスクリーンショット等に無断で使用すると即リジェクトとなります。
* AppleのシステムUI（例えばApple Payのロゴマーク等）の使い方がガイドラインに沿っていない場合もリジェクトされます。

### 🛡 対策方針
* アプリ内の文言、App Storeの宣伝文句において他社競合の名前を出さない。
* Apple Pay関連のマークを使う場合は、Apple公式の [Apple Pay Identity Guidelines](https://developer.apple.com/apple-pay/marketing/) を厳守します。
