## 1. テスト方針

STでは、ユーザー視点でのエンドツーエンド（E2E）操作を検証する。
シミュレータまたは実機でアプリを起動し、画面操作→データ結果を目視確認する。

---

## 2. テスト環境・前提条件

- **テスト方法**: Xcodeのシミュレーター（iPhone 16 Pro等）または実機にアプリをビルドして実行
- **データのリセット**: まっさらな状態からテストできるよう、アプリを一度削除（またはシミュレータ初期化）して再インストールしてから始める
- 初期データとして全体予算250,000円とカテゴリ別の予算がいくつか設定されている状態からスタート

---

## 3. テストケース

### 3.1 支出の追加・予算の連動確認（基本フロー）

一番重要な「お金を使ったら全体予算とカテゴリ予算が正しく減るか」のテストです。

- **[ ] ケースST-001**: (テスト観点: TP-10, TP-19)
    1. ホームのダッシュボードで、現在の「今月の残り予算（一番上の数字）」と「食費」カテゴリの残高をメモする
    2. 「記録をつける」ボタンから入力モーダルを開く
    3. カテゴリ「食費」を選択し、金額を `500` と入力し「使う」をタップ
    4. **[期待結果]**:
        - モーダルが閉じる
        - ホームの「今月の残り予算」が 500円 減っている
        - リストにある「食費」の残り予算が 500円 減っている

### 3.2 立替機能の確認（2段入力）

飲み会などの立替分が、自分の予算を圧迫しないかをテストします。

- **[ ] ケースST-002**: (テスト観点: TP-12, TP-20)
    1. 入力モーダルを開く
    2. 上部の「立替」トグルをONにする（デザインがオレンジに変わる）
    3. 上段の「みんなの立替分」に金額 `5000` を入力し、下段の「自分自身の支出」をタップし `1000` を入力
    4. 「立替」をタップ
    5. **[期待結果]**:
        - ホームの「今月の残り予算」が1000円のみ減っている（5000円全額は減らない）
        - 直近の履歴に自分の支出（1000円）が表示されている
        - カテゴリを選択していた場合、そのカテゴリの残高も1000円のみ減っている

- **[ ] ケースST-003**: (テスト観点: TP-03, TP-21)
    1. 入力モーダルを開き、立替トグルON
    2. 上段に金額 `3000` を入力し、下段の自分の支出欄は未入力（`0` のまま）で「立替」タップ
    3. **[期待結果]**:
        - アラート「自分の支出額を入力してください」が表示され、確定できない

- **[ ] ケースST-004**: (テスト観点: TP-03, TP-21)
    1. 入力モーダルを開き、立替トグルON
    2. 上段に `1000`、下段に `3000` を入力して「立替」タップ
    3. **[期待結果]**:
        - アラート「立替総額が自分の支出額より小さくなっています」が表示され、確定できない

### 3.3 履歴の編集・更新確認

入力した金額やカテゴリが間違っていた場合に、新しく追加されず「上書き」されるかのテストです。

- **[ ] ケースST-005**: (テスト観点: TP-13)
    1. ホーム画面の大きな円グラフをタップして「今月の全体予算」詳細を開く、またはメニューから「支出履歴を開く」
    2. 先ほど入力した食費 `500` 円の履歴行をタップする
    3. 編集モーダルが開かれたら、金額を `1000` に修正して「更新」をタップ
    4. **[期待結果]**:
        - 新しい履歴が追加されない
        - 該当の履歴行の金額が `1000` 円に変わっている
        - ホームの全体予算・食費予算から合計1000分のみ引かれた状態（追加で500円分のみ変動）

- **[ ] ケースST-006**: (テスト観点: TP-14)
    1. 支出履歴から、どれか一つの支出（例: 食費 1000円）を **左にスワイプ** し、「削除」をタップ
    2. **[期待結果]**:
        - 履歴一覧から該当の行が消える
        - ホームの「全体予算」と「カテゴリ予算（食費）」にその金額が **戻っている**（増えている）

### 3.4 臨時収入の確認

給与やプレゼントなどの臨時収入を記録するテストです。

- **[ ] ケースST-007**: (テスト観点: TP-09)
    1. ダッシュボードの全体予算をメモする
    2. 入力モーダルを開き、上部のセグメントで「臨時収入」を選択する（デザインが緑に変わる）
    3. 「給与」を選び、金額に `50000` を入力して「追加」をタップ
    4. **[期待結果]**:
        - 全体の「今月の残り予算」が 50000円 **増えている** こと
        - 臨時収入は予算総額（totalAmount）に加算される仕様

### 3.5 月次レビューと借金繰越の確認

月を跨いだときのリセット処理のテストです。

- **[ ] ケースST-008**: (テスト観点: TP-16)
    1. サイドメニューから「月末振り返り」をタップする
    2. 現在の実際の支出額と予算が正しいか確認する
    3. 「報酬を受け取って次月へ」ボタンをタップする
    4. **[期待結果]**:
        - 月がリセットされ、ダッシュボードの表示が次の月（基本予算250,000円など）に戻る
        - もし前月オーバーしていた場合、あらかじめ設定予算からそのオーバー分（借金）がマイナスされた状態でスタートしている

### 3.6 予算オーバー警告の確認

- **[ ] ケースST-009**: (テスト観点: TP-22)
    1. 入力モーダルから、今月の全体予算を超える支出を記録（例: 200万円を食費として記録）
    2. ホーム画面上部に「予算をオーバーしています！」というテキストが表示されていることを確認する
    3. **[期待結果]**:
        - （確認ボタンなどはなく）予算をオーバーしている旨だけが表示される

- **[ ] ケースST-010**: (テスト観点: TP-22)
    1. 前月の予算を超えているが、当月の予算は設定内の場合を再現する
    2. ホーム画面に「前月の借金により予算修正が必要です」等のテキストが表示されることを確認する
    3. **[期待結果]**:
        - 当月の予算がオーバーしているときには、来月の借金に関する警告は表示されない（当月オーバー警告が優先）

## 5. ユーザビリティ確認

TISカタログ：ユーザビリティテスト に準拠

### 5.1 入力方式と直感性

- **[ ] ケースST-016**: (テスト観点: No.57, No.73, No.74)
    1. 入力モーダルの各モード（支出/立替/臨時収入）を開く
    2. **[期待結果]**:
        - 全モードで標準キーボード（TextField等）が表示されず、カスタムキーパッドのみで入力できること
        - （ADHDヒューリスティック）文字入力の煩わしさがなく、直感的に数字だけ叩けること

### 5.2 ボタン文言・用語の一貫性

- **[ ] ケースST-017**: (テスト観点: No.79, No.80)
    1. 各モードでのボタン文言および画面上のラベルを確認する
    2. **[期待結果]**:
        - 支出モード: 確定ボタン「使う」
        - 立替モード: 確定ボタン「立替」、ラベル「自分の支出」「みんなの立替分」
        - 臨時収入モード: 確定ボタン「追加」
        - 編集モード: 確定ボタン「更新」
        - これらの文言がユーザーにとって迷いなく理解できること

### 5.3 ゼロ件・初期表示状態のUX

- **[ ] ケースST-018**: (テスト観点: No.13, No.14, No.20)
    1. インストール直後（データ0件）の状態でダッシュボードと履歴画面を開く
    2. 「記録をつける」で入力モーダルを開く
    3. **[期待結果]**:
        - 履歴リストが「まだ記録がありません」等の分かりやすい空状態表示になること
        - 入力モーダルが「支出モード」「金額0」「初期カテゴリ（食費等）選択済」で開くこと

### 5.4 ナビゲーションと完了後UX

- **[ ] ケースST-019**: (テスト観点: No.76, No.77)
    1. サイドメニューを開き、ダッシュボード、履歴、設定などの各画面に遷移する
    2. 支出を1件記録する
    3. **[期待結果]**:
        - メニューからの遷移がスムーズで迷わないこと
        - 支出記録後、自動でモーダルが閉じダッシュボードに戻り、すぐ結果が目視確認できること
