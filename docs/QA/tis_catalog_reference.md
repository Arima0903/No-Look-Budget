## 1. カタログの構造と読み方

### 1.1 カタログの列定義

| 列名 | 説明 |
|------|------|
| **大項目** | テスト対象の大分類（例: バリデーション、DB処理） |
| **中項目** | 具体的な検証領域（例: 入力チェック、登録） |
| **小項目** | 検証の切り口（例: 必須、文字種、桁数） |
| **観点** | 「○○であること」形式の期待結果文。`[]` 内は可変部分で、プロジェクトの設計書に読み替える |
| **カウント対象** | テストケース数の見積りに使う要素（例: 入力項目数、画面数） |
| **インプット成果物** | テスト設計に必要な上流工程の成果物 |

### 1.2 本アプリでの読み替え

| カタログ記載 | No-Look-Budget での読み替え |
|-------------|--------------------------|
| システム機能設計書 | 基本設計書 (BD-001) / 詳細設計書 (DD-001) |
| 画面機能要件定義 | 要件定義書 (REQ-001) の機能要件 |
| 方式設計書 | 基本設計書 §1（システム構成） |
| 共通コンポーネント設計書 | 詳細設計書 §1.2（TransactionService） |

## 2. 適用するテスト種別一覧

| テスト種別 | 適用 | 理由 |
|-----------|:----:|------|
| 機能テスト【共通】 | ✅ | バリデーション、DB（SwiftData）処理、計算処理が該当 |
| 機能テスト【モバイルアプリケーション】 | ✅ | iOSネイティブアプリのため直接該当 |
| 業務シナリオテスト | ✅ | 月次サイクル・立替フロー等の業務フローあり |
| セキュリティテスト | △ | ローカルアプリだが、Keychain・App Group保護の確認が必要 |
| 性能テスト | △ | サーバーなしだが、レスポンスタイムと大量データの確認が必要 |
| ユーザビリティテスト | ✅ | ADHD対応設計のためUX品質が特に重要 |
| 構成テスト | △ | iOS 17+のみだが、複数デバイスサイズの確認が必要 |
| **障害テスト** | ❌ | サーバーなし・ローカルアプリのため対象外 |
| **運用シナリオテスト** | ❌ | サーバー運用なしのため対象外 |
| **データ互換性テスト** | ❌ | 外部データ連携なしのため対象外 |
| **ストレス/ボリューム/ロングラン** | ❌ | ローカルアプリ・少量データのため対象外 |

## 3. 機能テスト【共通】から抽出した重要観点

| 2 | | 必須入力項目が入力済の場合、エラーとならないこと | 正常系テスト |
| 5 | 有効値 | 有効範囲内の値がエラーとならないこと | 金額 > 0 (VLD-001) |

| 7 | | 有効範囲の下限値-1でエラーとなること | 金額 = 0 の拒否 |
| 8 | | 有効範囲の上限値+1でエラーとなること | 16文字目の入力拒否 |
| 10 | 文字種 | 許容されない文字種ではエラーとなること | 計算式への不正文字入力 |
| 14 | 数値範囲 | 有効範囲内でエラーとならないこと | 正の整数金額の受付 |
| 16 | | 数値項目に数値以外が入力された場合のエラー | カスタムキーパッドで根本防止（設計判断） |

## 5. 業務シナリオテスト（カタログ p.25-26）

| カタログNo | 中項目 | 観点（原文要約） | 本アプリでの適用 |
|-----------|--------|----------------|----------------|
| 1 | 運用サイクル(通常) | 通常行われるシナリオに基づいた操作で正常動作すること | 日次: 支出記録→残高確認 |
| 2 | 運用サイクル(変則) | 変則的なシナリオでも正常動作すること | 月跨ぎ・借金繰越 |
| 3 | 運用サイクル(エラー) | エラー時のシナリオでも正常動作すること | バリデーションエラー後の再入力 |

**シナリオ検討のポイント（カタログ p.26 補足）:**
- 日次 / 週次 / 月次 / 年次 のサイクル
- 運用サイクルに休日が含まれる場合
- 日や月をまたぐ場合 → **月次レビュー・借金繰越のテストで重要**
