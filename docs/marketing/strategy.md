# No-Look-Budget マーケティング戦略（Draft）

## 1. ペルソナ（ターゲットユーザー）とインサイト

### 基本属性
* **メインペルソナ**: 開発者自身（浪費家、ADHD傾向、面倒くさがり）

### 既存の家計簿アプリ（マネーフォワード、Zaim等）への不満・挫折理由
1. **「開く」というアクションへの高い壁**
   * アプリを開かないと情報が確認できないため、そもそも面倒くさい。
   * ウィジェット機能はあるが、「総額」しか出ず「カテゴリ（予算）ごとに今いくら使っていて、残りいくら余裕があるのか」がパッと見で把握できない。結果、「一番知りたい情報」に辿り着くまでの手数が多すぎる。
2. **自動化の罠によるモチベーション崩壊**
   * クレジットカード連携は便利そうに見えて、定期的に連携が切れるため再設定が手間で放置する。
   * OCR（レシート撮影機能）があっても、その「撮影する」というアクションすらやらない。
3. **「立替（フロント）」による家計簿のノイズ**
   * 飲み会などで立替をした際、クレジットカードの請求額が一気に跳ね上がり、自分の本当の消費額（家計簿）のバランスが崩れる。通常の家計簿アプリではこれを修正・分離するのが非常に手間で、これが原因で管理を放棄してしまう。

### ユーザーの究極の欲求（Core Desire）
* **「自分が今、何にいくら使って、これ以上いくら使えるのか（残りの余裕）」を、一切の手間なく、一瞬で把握したい。**

## 1. ペルソナ（ターゲットユーザー）とインサイト（続き）

### 手段の壁と「入力」の限界
* 現状は「自分がいくら使ったか全く把握していない」状態。
* 音声入力などは場所を選ぶため現実的ではない。
* **許容できる入力ハードル**: スマホを開いた瞬間に（ホーム画面/ロック画面などで）「現在の残高・予算」が常に目に入る状態であり、その数字をタップするだけで直接修正（入力）できるレベルのシームレスさが必要。

### 「立替」問題の真因：ADHD的完璧主義の崩壊
* 立替によって家計簿の金額が跳ね上がる（マイナスになる）こと自体が嫌なのではない。
* 「それまで綺麗に記録できていた自分のルール・数字」が、他人の要因（立替）によって一気にノイズだらけになり、**「完璧な状態」が崩れることで一気にやる気を無くす（＝ゼロか百かの思考）**のが最大の原因。

### 「No-Look」の本質：回避ではなく「忘却」への対策
* お金に向き合いたくない（心理的ダメージからの回避）わけではない。単に「入力・確認する習慣がないから忘れる」だけ。
* むしろ浪費家には生半可な見せ方では響かないため、**「今いくら使っていて、ヤバい状態なのか」を強制的に視界に入れ、痛烈に現実を突きつける**強いフィードバックが必要。

## 2. 競合優位性とポジショニング（Where to play）

### 競合アプリ（マネーフォワード、Zaim等）の現在地
* **誰向けか**: 「几帳面な人」「お金の流れを細かく分析したい人」
* **主な提供価値**: 銀行・クレジットカードの完全自動連携、レシートOCRによる高精度な費目仕分け。
* **弱点（当ペルソナにとって）**: 初期設定の壁が高い。連携エラーが起きると修復が手間。画面を開かないと状況が分からず、「今月あといくら使えるか」という一番知りたい情報へのアクセスが遠い。

### No-Look-Budget のポジショニング
* **誰向けか**: 「浪費家」「ADHD傾向で継続が苦手な人」「細かい費目はどうでもいい人」
* **主な提供価値（コア機能）**:
    1. **強制視界占有（No-Look Experience）**: ロック画面/ホーム画面のダイナミックウィジェット等で、「今月の残り予算」をアプリを**開くことなく**、強烈なビジュアル（赤字、警告色など）で常時見せつける。
    2. **1タップ入力**: ウィジェットをタップした瞬間に金額入力画面が立ち上がり、費目選択などは極力省き、最速で残高を減らす体験。
    3. **立替ノイズキャンセリング（完璧主義崩壊の防止）**: 飲み会等の立替を「一時的なバッファ」としてスワイプ等で即座に別枠へ退避させ、メインの「自分の予算」を綺麗な状態に保つ。

## 3. MVP（Minimum Viable Product）の機能要件

競合の「何でもできる」に勝つためには、「これしかできないが、これに関する体験は圧倒的」という引き算のUXが必要です。

**【MVPで実装すべき機能（Must Have）】**
1. **残高（残り予算）の超巨大表示**: アプリを開いた瞬間に「あと〇〇円」だけが目に入るUI。
2. **ダイナミックウィジェット（iOS）**: ロック画面・ホーム画面への強制配置。残高が減るごとに色が劇的に変わる（例：黒字→黄色→血のような赤）。
3. **最速入力UI**: ウィジェットタップ → 金額数字ボタン → 「使った」ボタンのみ。（費目タグは後回し、または最小限の選択制）
4. **立替セパレーター**: 入力時に「立替スワイプ」をすると、その金額が別枠（回収待ちリスト）に移動し、メイン予算の残高には影響を与えない（あるいは立て替えた分は別予算としてプールされる）機能。

**【MVPでは実装しない機能（Won't Have）】**
* 銀行・クレジットカードの自動API連携（エラーによる離脱を防ぐため、あえて手動に絞る）
* レシート撮影（OCR）機能
* 細かすぎる費目カテゴリ（食費・交際費・その他 程度で十分）
* 複雑なグラフ分析

## 4. マネタイズ戦略（段階的アプローチ）

ターゲットである「浪費家ユーザー」に継続的に使ってもらいつつ、収益化を図るための戦略。

### Phase 1: MVPリリース時（完全無料・圧倒的体験の提供）
まずは「No-Look Experience」と「立替セパレーター」のコアな価値を無料で全て解放し、**「これなら続く！」という感動体験（Aha Moment）** とアクティブユーザー（DAU）の獲得に全振りする。
* **見せつけUI（ウィジェット）**: 無料で制限なく提供。本アプリの目玉であり、これがないと成立しないため。
* **立替セパレーター**: これも無料で無制限提供。完璧主義を維持するための必須機能。
* **広告**: MVP段階では一切なし（UIの圧倒的なクリーンさをアピールする）。

### Phase 2: ユーザー定着後のマネタイズ導入
ユーザーが「このアプリがないと浪費してしまう」と依存し始めたタイミングで収益化を開始。

1. **フリーミアムモデル（広告モデルの導入）**
   * 無料版に広告を導入する。ただし、「入力を阻害する（広告を見ないと入力できない）」ような体験の悪化は避ける。あくまで画面下部のバナーや、予算振り返り画面でのネイティブ広告など。
2. **サブスクリプション（プレミアム版: 広告非表示＋α）**
   * 月額課金（例: 300円〜500円）で広告を完全非表示に。
   * **＋αの価値**: 過去データの無制限閲覧、より高度なテーマカラーのアンロックなど。
3. **将来構想：「罰金 / 強制ロック」機能（行動経済学アプローチ）**
   * 今後、より強力な自制を求めるユーザー向けに、「予算オーバーでペナルティ（課金）」といった尖った機能をプレミアムオプションとして提供する余地を残す。

## 5. MVP（Minimum Viable Product）の機能要件（改訂版）

**【MVPで実装すべき機能（Must Have）】**
1. **残高（残り予算）の超巨大表示**: アプリを開いた瞬間に「あと〇〇円」だけが目に入るUI。
2. **ダイナミックウィジェット（iOS）**: ロック画面・ホーム画面への強制配置。残高が減るごとに色が劇的に変わる。
3. **最速入力UI**: ウィジェットタップ → 金額数字ボタン → 「使った」ボタンのみ。
4. **立替セパレーター**: スワイプで立替金を別枠（回収待ちリスト）へ退避させる機能。
5. **Apple Wallet (Apple Pay) 連携【重要追加】**:
   * メインの決済手段（ウォレット登録のクレカ・Suica等）で支払った際に、**自動で家計簿に金額が即時反映される**機能。（ユーザーが最も嫌がる「手入力の労力」をシステム側で極限まで削減する）

**【MVPでは実装しない機能（Won't Have）】**
* 銀行口座や、Apple Wallet非対応のクレジットカードとのAPI連携（マネーツリーや各種銀行APIを利用した従来型のスクレイピング連携は、エラーによる離脱を防ぐため実装しない）
* レシート撮影（OCR）機能
* 細かすぎる費目カテゴリ
* 複雑なグラフ分析
