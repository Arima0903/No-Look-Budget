## 1. フィードバック収集の仕組み

### 1.1 App Store レビュー監視

**運用方法**
- App Store Connect の「レビュー通知」を開発者メールに設定し、新規レビューを即時把握
- 毎週月曜日に週次レビュー集計を実施（件数・評価平均・頻出キーワード）
- 低評価レビュー（1〜2星）は24時間以内に返信を試みる

**レビュー分類タグ（定性分析用）**
```
[クラッシュ] - アプリが落ちる
[UI/UX]     - 操作感・デザインへの不満
[機能要望]  - 新機能のリクエスト
[バグ]      - 動作不良（クラッシュ以外）
[ウィジェット] - ウィジェット固有の問題
[課金]      - プレミアム・価格への不満
[称賛]      - ポジティブなフィードバック
```

**ツール候補**
- App Store Connect（無料・公式）
- AppFollow または ReviewBot（Slack連携で即時通知、将来導入検討）

## 2. 優先度付けフレームワーク（RICEスコア）

### 2.1 RICE スコアの定義

```
RICE スコア = (Reach × Impact × Confidence) / Effort

Reach（影響ユーザー数）: その機能を使う可能性のあるユーザー数（週次）
Impact（インパクト）: 1人のユーザーへの影響度（1=小、2=中、3=大、5=特大）
Confidence（確信度）: 見積もりの確かさ（%）
Effort（工数）: 実装に必要な人月（小数点可）
```

### 2.2 評価例（実際の運用例）

| 機能/バグ | Reach | Impact | Confidence | Effort | RICE スコア |
|---------|-------|--------|-----------|--------|------------|
| ウィジェット更新遅延バグ修正 | 1000 | 5 | 90% | 0.5 | 9,000 |
| AIファイナンシャルプランナー | 200 | 5 | 70% | 3.0 | 233 |
| iCloud同期 | 500 | 3 | 80% | 2.0 | 600 |
| カスタムカテゴリアイコン | 300 | 2 | 90% | 1.0 | 540 |
| 年間グラフ分析 | 150 | 2 | 70% | 1.5 | 140 |

**優先度の判断基準**
- RICE スコア 1,000以上: 最優先（次のスプリントへ即投入）
- RICE スコア 500〜999: 高優先（次々スプリントで対応）
- RICE スコア 100〜499: 中優先（バックログに積む）
- RICE スコア 100未満: 低優先（将来検討）

### 2.3 ショートカット判断（緊急度の考慮）

RICEスコアに加え、以下は即時対応とする（スコア計算をスキップ）。
- アプリ起動不能・主要機能が動作しないクラッシュ
- App Store審査却下につながるコンプライアンス違反
- セキュリティ脆弱性（個人データ漏洩リスク）

## 3. スプリント計画（2週間サイクル）

### 3.1 スプリントの基本構造

```
スプリント期間: 2週間（月曜開始・日曜終了）

Week 1:
  月: スプリント計画（バックログ確認・タスク選定）
  火〜金: 開発・実装
  土: セルフコードレビュー + 単体テスト

Week 2:
  月〜水: 実装継続 + 統合テスト
  木: シミュレータでの動作確認
  金: TestFlightビルド作成・内部テスト
  土: 振り返り（レトロスペクティブ） + 次スプリント準備
  日: バッファ（修正対応）
```

### 3.2 スプリントのインプット

**バックログアイテムのソース**
1. App Storeレビュー（前週分の集計結果）
2. SNSメンション（ユーザーの自然な声）
3. アプリ内フィードバックフォーム
4. Crashlytics のクラッシュレポート
5. 開発者自身の使用感による気づき

**スプリント計画会議（月曜2時間）のアジェンダ**
```
1. 前スプリントのKPIレビュー（DAU・継続率・レビュー評価）: 30分
2. バックログのRICEスコア更新: 20分
3. 今スプリントのタスク選定（工数合計: 最大20h/2週間）: 40分
4. 完了条件（DoD）の確認: 20分
5. リスク確認: 10分
```

### 3.3 スプリントの完了条件（Definition of Done）

- [ ] 実装コードがビルドエラー・警告ゼロ
- [ ] 新機能に対応するXCTestが追加・全件Pass
- [ ] シミュレータでの動作確認（iPhone 15 Pro / SE3 の2端末）
- [ ] SwiftLintの警告ゼロ
- [ ] CHANGELOG.md の更新
- [ ] TestFlightへのビルドアップロード完了

### 3.4 振り返り（レトロスペクティブ）

毎スプリント最終土曜に以下を記録する。

```markdown

- [x] ...

- [ ] ... （理由: ...）

- 新規インストール: xxx件
- App Store評価: x.x

### Keep（続けること）
- ...

### Problem（問題だったこと）
- ...

### Try（次スプリントで試すこと）
- ...
```

## 4. バージョン管理ポリシー

### 4.1 セマンティックバージョニング（MAJOR.MINOR.PATCH）

```
MAJOR（例: 2.0.0）: 破壊的変更・アーキテクチャ刷新・UI全面リニューアル
  条件:
  - データモデルのマイグレーションが必要な変更
  - ユーザーが既存の操作方法を覚え直す必要がある変更
  - 既存機能の大幅な削除・変更

MINOR（例: 1.3.0）: 後方互換性を維持した新機能追加
  条件:
  - 新しいプレミアム機能の追加
  - 新しいカテゴリ・ウィジェットタイプの追加
  - AIプランナーなど大型機能のリリース

PATCH（例: 1.2.3）: バグ修正・軽微な改善
  条件:
  - クラッシュ修正
  - UI表示のズレ修正
  - パフォーマンス改善
  - テキスト・コピーの修正
```

### 4.2 リリースサイクルの目安

| リリース種別 | 頻度の目安 | 審査時間の見込み |
|------------|----------|--------------|
| MAJORリリース | 半年〜1年に1回 | 1〜3日（要事前審査依頼） |
| MINORリリース | 4〜6週に1回（2〜3スプリント） | 1〜3日 |
| PATCHリリース | 随時（緊急バグ対応は即日提出） | 24時間以内（緊急審査申請） |

### 4.3 CHANGELOG.md の管理ルール

```markdown
## [1.3.0] - 2026-XX-XX
### 追加
- AIファイナンシャルプランナー機能（プレミアム）
- iCloud同期機能（プレミアム）

### 変更
- カテゴリ上限を無料版6個に変更

### 修正
- ウィジェット更新が遅延する問題を修正
```

## 5. クラッシュ/バグ対応フロー

### 5.1 重大度レベルの定義

| レベル | 定義 | 対応期限 |
|--------|------|---------|
| P0（緊急） | アプリ起動不能・主要機能が全滅 | 4時間以内に修正開始 |
| P1（高） | クラッシュ率 1%以上・データ消失リスク | 24時間以内に修正着手 |
| P2（中） | 特定操作でクラッシュ・主要機能の動作不良 | 次のPATCHリリース（1週間以内） |
| P3（低） | UIのズレ・軽微な表示不具合 | 次のMINORリリース |

### 5.2 バグ対応フロー（P0/P1）

```
1. 検知
   ├── Crashlytics アラート受信 or ユーザーレポート
   └── 重大度レベルを判定（P0/P1/P2/P3）

2. 調査（P0: 即時、P1: 1時間以内）
   ├── Crashlytics のスタックトレース確認
   ├── 再現手順の特定（シミュレータ + 実機）
   └── 影響範囲（ユーザー数・機能）の確認

3. 修正
   ├── 修正ブランチ作成: hotfix/YYYY-MM-DD-バグ概要
   ├── 修正実装 + ユニットテスト追加
   └── コードレビュー（自己レビュー + テスト全件通過確認）

4. リリース
   ├── TestFlight でのビルド検証（30分以上テスト）
   ├── App Store に緊急審査申請（Expedited Review）
   └── App Store 承認後、即時リリース

5. ポストモーテム
   ├── 発生原因・対応内容・再発防止策を docs/QA/ に記録
   └── 同様のバグを防ぐユニットテストの追加確認
```

### 5.3 バグレポートテンプレート（docs/QA/ 保存）

```markdown

**重大度**: P0 / P1 / P2 / P3
**発見日**: YYYY-MM-DD
**修正バージョン**: x.x.x
**ステータス**: 調査中 / 修正中 / レビュー待ち / クローズ

### 期待する動作
...

### 実際の動作
...

...

...
```

## 6. A/Bテスト戦略

### 6.1 A/Bテストの基本方針

- 一度に実施するA/Bテストは最大2件まで（結果の混在防止）
- テスト期間: 最低2週間（統計的有意性の確保）
- サンプルサイズ: 各グループ最低100人（DAU 600人以上になってから本格実施）

### 6.2 優先するA/Bテスト候補

| テスト名 | 変数A（コントロール） | 変数B（テスト） | 測定指標 |
|---------|-------------------|--------------|---------|
| App Storeスクショ1枚目 | ウィジェット赤警告画面 | 残高大きく表示 | インストール転換率 |
| プレミアム誘導タイミング | 初回起動翌日 | D14（2週間後） | 課金転換率 |
| プレミアム価格 | 月額480円 | 月額380円 | LTV（課金転換率×継続率） |
| ウィジェット残高表示 | 絶対額（円） | 残り%表示 | D7継続率 |
| MonthlyReview レビュー依頼 | 振り返り完了直後 | 翌日朝の通知 | レビュー投稿率 |

### 6.3 A/Bテストの実装方針

**実装方法（iOSネイティブ）**
```swift
// Firebase Remote Config を使用（無料枠で十分）
// 起動時にパラメータを取得し、フラグで分岐

enum ABTestVariant: String {
    case control = "control"
    case treatment = "treatment"
}

// RemoteConfig から取得して分岐
let variant = RemoteConfig.remoteConfig()["premium_cta_timing"].stringValue
```

**将来の選択肢**
- Firebase Remote Config（無料・推奨）
- Statsig（無料プラン有、より高度な管理）

### 6.4 テスト結果の評価基準

```
成功条件:
- 測定指標の改善率: +10%以上
- p値: 0.05以下（統計的有意）
- 信頼区間の下限: 0%以上（悪化しない）

判断フロー:
  改善かつ統計的有意 → 変数Bを本採用
  改善だが統計的に不十分 → テスト期間を延長
  差なし or 悪化 → 変数A（コントロール）を維持 + 仮説を見直す
```

### 6.5 学習の蓄積

テスト結果は `docs/QA/ab_test_results.md` に記録し、以下を蓄積する。
- 何を試したか
- 結果と解釈
- 次に試すべき仮説
