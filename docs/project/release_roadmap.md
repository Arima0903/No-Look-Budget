## Phase C: MVP 公開・ソフトローンチ（2026/4末〜5月）

### ソフトローンチ戦略

審査通過後は段階的に告知を行い、初動レビューを獲得する。

**告知チャネル（優先順位順）**

1. **Twitter/X**: リリース当日に動画ツイート（ウィジェットの動作デモ）
2. **個人ブログ / Zenn**: 開発過程の技術記事と同時に紹介
3. **Product Hunt**: 英語ページを用意して国際向けに投稿（任意）
4. **身近な人への告知**: 家族・友人への直接案内でレビュー 5件以上を確保

### App Store レビュー獲得施策

- アプリ内で `SKStoreReviewRequest` を適切なタイミングで呼び出す
  - タイミング: 3回目の支出入力後 / 月またぎ完了後
- レビュー依頼のタイミングはユーザーが「成功体験」を得た直後に限定

### KPI モニタリング開始

- App Store Connect の Analytics でダウンロード数・インプレッション・転換率を週次確認
- セッション時間・クラッシュ率を Xcode Organizer でモニタリング

## Phase D: フィードバックスプリント（2026/5〜6月）

### 初動レビュー対応

- App Store レビューを毎日確認し、1営業日以内に返信
- クラッシュ報告は **48時間以内** にパッチリリース

### 優先フィードバック項目のバックログ化

フィードバックを以下の軸でトリアージする:

| 優先度 | 基準 | 対応方針 |
|---|---|---|
| P0（即対応） | クラッシュ・データ消失・入力不可 | ホットフィックス |
| P1（v1.1 対象） | 複数ユーザーから同一要望・UX 改善 | 次スプリントで実装 |
| P2（検討） | 単発要望・Phase E 以降の機能 | バックログ積み |

### v1.1 リリース計画

- 目標: Phase D 終了までに 1回はアップデートリリース
- 内容: バグ修正 + ユーザー要望の小規模 UX 改善
- AdMob 事前申請: Phase D 期間中に AdMob アカウントを先行登録し、審査を通過させておく

## リスク・バッファ表

| リスク | 影響度 | 発生確率 | 対策 |
|---|---|---|---|
| App Store 審査リジェクト | 高 | 中 | 初回申請前に審査ガイドライン（App Review Guidelines）を再確認。メタデータと実際の機能を一致させ、プライバシーポリシーを丁寧に記載する |
| TestFlight でのクラッシュ発見 | 中 | 中 | ST（手動テスト）で事前に主要フローを確認済みにしておく。クラッシュ発見時は申請前にホットフィックス |
| DAU が目標を下回る | 中 | 中 | SNS 投稿の継続・ASO 改善（キーワード最適化・スクリーンショット改善）。ターゲットユーザーのコミュニティへの参加 |
| AdMob 審査通過に時間がかかる | 低 | 低 | AdMob 登録と審査申請は Phase D 中に先行実施。審査通過後に Phase E 実装を開始 |
| StoreKit の審査リジェクト | 中 | 低 | 課金フローに関するガイドライン（Guideline 3.1.1）を事前確認。テスト環境での完全動作を確認してから申請 |
| ネガティブレビューの急増 | 高 | 低 | レビューへの迅速な返信と修正対応。P0 バグは 48時間以内にパッチリリース |
| SwiftData マイグレーション失敗 | 中 | 低 | Phase E でモデル変更が生じる場合は `VersionedSchema` と `MigrationPlan` を実装してからリリース |
